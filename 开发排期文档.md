# 日记记录平台 - 开发排期文档

## 1. 排期说明

本排期基于以下默认前提制定：

- 开发模式：单人开发
- 项目形态：本地优先 Web + Electron 桌面应用
- 技术栈：Node.js + Express + SQLite + 原生 HTML/CSS/JS
- 当前文档基线日期：`2026-04-26`
- 开发目标：先完成第一版 MVP，再进入体验优化

本排期按 5 周规划，起始日期建议为：`2026-04-27`

---

## 2. 开发目标

第一版必须完成以下主链路：

`写记录 -> 保存本地 -> 上传图片 -> 时间轴查看 -> AI提炼 -> 首页展示 -> 周/月总结查看`

第一版不强求复杂视觉和高级编辑器，优先保证：

- 数据稳定落盘
- 页面主流程可用
- AI 提炼结构化输出
- 首页信息架构成立

### 验收标准

- 首页能读取最近提炼结果

## 6. 模块拆分与依赖关系

## 6.1 依赖顺序

1. 数据库与后端接口
2. 页面骨架与主导航
3. 写记录页
4. 时间轴页
5. AI 日提炼
6. 首页模块
7. 周/月总结
8. 设置与备份
9. 打包与稳定性优化

## 6.2 关键阻塞项

- 数据表设计如果反复变动，会拖慢后续所有页面
- AI 提炼模板不稳定，会导致首页结构反复重写
- 图片存储路径如果不先定，会影响写记录、时间轴、备份

## 8. 风险与应对

| 风险 | 说明 | 应对方式 |
|------|------|----------|
| AI 输出不稳定 | 提炼字段不统一，影响首页展示 | 固定模板 + 固定 JSON 结构 |
| 本地图片管理混乱 | 文件路径、删除策略易出错 | 先统一上传目录与命名规则 |
| 页面范围膨胀 | 首页内容容易越加越多 | 严格以当前线框图为边界 |
| 排期被 UI 细节拖慢 | 早期过多打磨视觉 | 先完成功能，再做视觉优化 |
| 数据结构反复调整 | 导致接口和页面重写 | 第 1 周冻结核心表结构 |
